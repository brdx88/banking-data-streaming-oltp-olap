# Architecture Context

## End-to-End Flow

```text
Python Producers
    |
    v
Confluent Cloud Topics
    |
    +--> analytics consumer
    |
    +--> analytics datamart consumer
    |
    +--> fraud detection consumer
    |
    +--> data warehouse consumer
            |
            v
        BigQuery
```

## Event Domains

### 1. Mobile Banking Events

Represents digital channel activities from users interacting with the mobile app.

Examples:

- login
- logout
- password_reset
- balance_inquiry
- transfer_initiated
- bill_payment
- device_registered

### 2. Transaction Events

Represents money movement or financial activity.

Examples:

- transfer
- purchase
- atm_withdrawal
- cash_deposit
- refund
- reversal

### 3. Customer Service Events

Represents service and support interactions.

Examples:

- complaint_opened
- complaint_closed
- dispute_opened
- dispute_resolved
- card_block_request
- profile_update_request

## Topic Strategy

Use the Confluent Cloud topics that have already been created in the `bank-nusantara-streaming` environment on the `bank-nusantara-kafka` cluster.

Source topics:

- `mobile-banking-activity` with 3 partitions
- `transaction-events` with 3 partitions
- `cs-interactions` with 2 partitions

Optional downstream topics for derived outputs:

- `analytics-metrics` with 1 partition
- `fraud-alerts` with 1 partition
- `dw-dead-letter` with 1 partition

## Consumer Responsibilities

### `analytics.py`

- Read from source event topics
- Compute simple aggregates such as event counts, transaction volume, active users, complaint counts
- Optionally publish summarized metrics to `analytics-metrics`

### `analytics_datamart.py`

- Consume source topics
- Build real-time KPI aggregates in short windows
- Write KPI rows into BigQuery datamart tables

### `fraud_detection.py`

- Primarily consume transaction and mobile banking events
- Detect suspicious patterns such as:
  - high-value transfers
  - bursts of failed logins
  - rapid device changes
  - transactions in unusual locations
- Publish alerts to `fraud-alerts` and/or log them

### `data_warehouse.py`

- Consume selected events
- Normalize payloads to warehouse-ready rows
- Write to BigQuery raw or curated tables
- Handle failed inserts with retry and dead-letter strategy

## Recommended Processing Layers

### Raw Event Layer

Events as produced, with minimal transformation.

### Curated Event Layer

Validated and normalized rows with stable column names for analytics.

### Derived Insight Layer

Aggregates, KPIs, and fraud alerts generated by consumers.

### Datamart KPI Layer

Real-time KPIs written into BigQuery tables for dashboard consumption.

## Deployment Posture

### Local Runtime

- Producers and consumers run locally via Python scripts
- Secrets loaded from `.env`-style config
- Logging to console first

### Managed Services

- Kafka broker, topics, API keys: Confluent Cloud
- Analytics warehouse: BigQuery

## Non-Goals for First Version

- Exactly-once processing guarantees
- Complex stream processing framework
- Schema Registry + Avro/Protobuf on day one
- ML-based fraud scoring
- Full CI/CD automation before the pipeline works end-to-end
