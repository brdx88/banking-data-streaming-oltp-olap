# Implementation Roadmap

## Phase 1: Project Bootstrap

- Create repository folders
- Add `README.md`
- Add `requirements.txt`
- Add `.gitignore`
- Add `config/confluent.env.example`

Definition of done:

- Local environment can install dependencies
- Secret file strategy is clear

## Phase 2: Shared Foundation

- Implement `schemas/events.py`
- Implement `utils/kafka_client.py`
- Create reusable event serialization helpers

Definition of done:

- A producer can publish a valid JSON event to Confluent Cloud
- A consumer can subscribe and parse the event

## Phase 3: Producers

- Implement `producers/mobile_banking.py`
- Implement `producers/transaction.py`
- Implement `producers/customer_service.py`

Definition of done:

- Each producer can run independently
- Each producer emits realistic sample events with consistent schema

## Phase 4: Consumers

- Implement `consumers/analytics.py`
- Implement `consumers/analytics_datamart.py`
- Implement `consumers/fraud_detection.py`
- Implement `consumers/data_warehouse.py`

Definition of done:

- Analytics consumer computes basic metrics
- Datamart consumer writes real-time KPI aggregates into BigQuery
- Fraud consumer emits rule-based alerts
- Data warehouse consumer inserts records into BigQuery

## Phase 5: Validation

- Run end-to-end local testing
- Verify topic flow in Confluent Cloud UI
- Verify inserted data in BigQuery
- Verify fraud and analytics outputs

Definition of done:

- Event generated by producer is visible in Kafka
- At least one downstream consumer processes it successfully
- At least one row lands in BigQuery

## Recommended Build Order

1. `schemas/events.py`
2. `utils/kafka_client.py`
3. `producers/transaction.py`
4. `consumers/data_warehouse.py`
5. remaining producers and consumers

Reasoning:

- Transaction events are the most central business signal
- BigQuery sink gives immediate proof of value
- Other producers and analytical consumers can then reuse the same foundation

## Open Decisions To Finalize

- Will all events use one shared topic or one topic per domain?
- Will BigQuery store raw JSON only, or partially flattened fields too?
- Will fraud alerts be stored only in Kafka, or also persisted to BigQuery?
- Will the project use plain scripts only, or package entry points later?
